## 高優先（資料安全 / 流程正確性）

### H1. 無文件級安全，個資全可見 ✅
- **位置**：`AppProperties/database.properties`（ACL）、四表單（無 Readers/Authors 欄位）。
- **現況**：ACL 幾乎全員 Manager、無 Role 落實到資料列；四表單皆無 `Readers` 欄位。
- **影響**：任何能開啟資料庫者可讀寫**所有**申請（姓名/地址/緊急聯絡人/證件）、訪客、違規紀錄。
  「對學生自助開放」在此狀態下**不安全**。
- **建議修法**（獨立大案，需資料遷移＋測試）：
  1. 定義 Role：`[Admin]`/`[Staff]`（已在 database.properties 建好）＋ `[Student]`。
  2. 四表單加 `Readers` 欄位（如：`@Name([CN];@UserName) : approverName : "[Admin]" : "[Staff]"`）。
  3. 學生 ACL 由 Manager 降為 **Author**（含 Create），承辦人 Editor/以上。
  4. 申請表「申請人」存 NotesName，供 Readers/Authors 控管。
- **風險**：改 ACL/Readers 會立即影響可見性，須先在測試副本驗證，避免鎖死自己。

### H2. 確認簽核閉環完整（多數已具備，少數待補）✅🔍
- **位置**：四表單 Action + 對應「待辦」視圖。
- **已查證屬實**：
  - 違規 `違規紀錄表`：Action「完成處理」→ `processStatus:="已完成"`、「已撤銷」。✅
  - 訪客 `訪客登記表`：Action「核准」→ `visitStatus:="已核准"`、「拒絕」→`"拒絕"`、報到→`checkInTime:=@Now`、離開→`checkOutTime:=@Now; visitStatus:="已離開"`。✅
  - 視圖過濾全部正確一致（待審核/待核准/待處理皆對應正確狀態字串）。✅
- **待補/待確認**：
  - 🔍 `宿舍申請表`(DormApp，**base64**)：待審核視圖的「審核申請」鈕只做 `@Command([EditDocument])`，
    **真正核准動作在表單上**——需在 Designer 確認表單有 Action 寫
    `applyStatus:="已核准"` ＋ `reviewer` ＋ `reviewDate`（`已核准`視圖有這些欄，推測已具備）。
  - 簽核三表的**狀態變更皆未寫稽核時間戳到統一欄位**時，建議補 `@Now` 與審核人（部分已寫 reviewDate/approvedAt）。

### H3. 訪客「報到時間」會被重複覆蓋 ✅
- **位置**：`訪客登記表.form` Action 報到：`FIELD checkInTime := @Now;`（且 `checkInTime` 同時為 editable 欄）。
- **影響**：報到鈕被按第二次會用新 `@Now` **覆蓋原始入館時間**；可被竄改、稽核失真。
- **建議修法**：改為僅在空值時寫入——`FIELD checkInTime := @If(checkInTime=""; @Now; checkInTime);`
  並同步把 `visitStatus` 由「已核准」推進到「已入館/在館」（目前報到只寫時間、未推進狀態）。
  離開同理：應先確認已報到（`checkInTime!=""`）再允許 checkOut。
- **風險**：低（單一公式、可讀表單 FVisitor）。

## 中優先（一致性 / 維護性 / 防呆）

### M1. FRoom 欄位命名 snake_case，與全庫 camelCase 不一致 ✅
- **位置**：`房間資料表.form` 與 `VRoomLookup`/`VRoomAvailable`：`room_number`、`room_type`、
  `gender_limit`、`available_beds`、`updated_at`（snake_case）；其餘表單/視圖全 camelCase。
- **影響**：跨表閱讀/維護易混淆；新人或 AI 容易引用錯欄名。
- **建議修法**：統一改 camelCase（`roomNumber`/`roomType`/`genderLimit`/`availableBeds`/`updatedAt`）。
- **風險/前置**：**需資料遷移**（既有文件欄名要一併改，否則查無值），且 FRoom 內文有 base64；
  屬技術債，建議獨立小案、低急迫。

### M2. `updated_at` 計算模式需確認 🔍（記憶記載）
- **位置**：`房間資料表.form` `updated_at`（datetime）。
- **疑點**：純 Formula 下 occupancy/available_beds 只在 FRoom 開啟/存檔時重算並寫回索引，
  `VRoomAvailable`（`status="可入住"`）的篩選會有延遲（非即時）。
- **建議**：確認 `updated_at`/`available_beds` 的計算事件（Computed vs CWC vs Input Translation），
  必要時改由存檔時 Action 統一寫入；或文件化「非即時」這個已知限制。

### M3. @DbLookup/@DbColumn 防呆與輸入驗證盤點 🔍
- **位置**：各表單帶值欄位（學號→姓名/系級、房號→入住數）、申請/訪客/違規關聯欄。
- **建議**：確認每個 @DbLookup 都 `""\:"NoCache"` ＋ `@If(@IsError(...);"";...)` 包裝，
  且 key 欄位有 Input Validation（查無 key 擋下）。skill `formula-cookbook.md` §2/§4 有現成樣板。

### M4. 左欄大綱缺「系統首頁」回首頁入口 ✅
- **位置**：`ClientOutLine`（現為 base64）：解碼後條目＝VRoom→DormApp(群)→FVisitor(群)→FViolation(群)，
  **無 PHome 條目**。
- **影響**：進到任一視圖後，左欄沒有「回系統首頁」的項目（右框被視圖佔據後不易回首頁）。
- **建議修法**：在 Designer 的大綱最上方加一條 `系統首頁`（Page=PHome，Target frame=Content）。詳見導航設定步驟。

## 低優先 / 資訊

### L1. PHome 內文匯入亂碼（本輪擱置）✅
- **現象**：DXL importer 對「長段落混排 CJK＋英文（Lookup/@DbLookup/roomKey…）」的 richtext 會在較後段
  產生 mojibake。Hero/統計卡/短標題正常。
- **建議**：未來修復時，把長段落**拆成更短的純文字 run**、或英文代號移到括號/另起短行，降到 importer
  可接受複雜度；或於 Designer 直接以 rich text 編輯器輸入該段。

### L2. 三顆管理快捷鈕的角色隱藏待補 ✅
- **位置**：`系統首頁.page` actionbar 待審核/待核准/待處理三鈕（目前一律顯示）。
- **原因**：Page 的 action **不支援** `hide='formula'`/`<hidewhen>`（importer 報錯，已移除）。
- **建議**：在 Designer 為這三顆 action 的「Hide」頁籤填公式 `!@IsMember("[Admin]":"[Staff]"; @UserRoles)`。
